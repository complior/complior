# ADR-002: 3-уровневая система памяти

**Статус**: Superseded — memory system removed in v8 (LLM chat removed, daemon architecture replaces session context)
**Дата**: 2026-02-17
**Автор**: Marcus (CTO)

## Контекст

Coding assistant должен помнить контекст проекта между сессиями, управлять контекстным окном LLM, и иметь доступ к обширной базе знаний о регуляциях и AI-инструментах, не перегружая промпт.

Существующие подходы:
- **OpenCode**: единый `.md` файл — плоский, без структуры, ограниченный
- **Claude Code**: `CLAUDE.md` — инструкции проекта, но не compliance-знания
- **Cursor**: `.cursorrules` — правила, но без персистентной памяти проекта

## Решение

3-уровневая система памяти, оптимизированная для контекста комплаенса:

### Уровень 1: Память проекта (персистентная, автообновляемая)

`.complior/memory.json` — автоматически обновляется после каждого скана, фикса и сессии. Читается LLM-инструментами через `getProjectMemory()` — НЕ загружается в системный промпт (экономия контекстного окна).

Содержит: профиль проекта (фреймворк, AI-инструменты, юрисдикции, уровень риска), история score, примененные фиксы, итоги последней сессии, ожидающие действия.

### Уровень 2: Контекст сессии (скользящее окно + суммаризация)

- Во время TUI-сессии: полный диалог в контекстном окне LLM
- При приближении к лимиту (~80% окна модели): авто-суммаризация старых сообщений в 200-токенную сводку
- Сохранение: последние 5 сообщений + контекст текущей задачи + профиль проекта
- Полная история в SQLite (better-sqlite3) для истории сессий

### Уровень 3: Знания комплаенса (по запросу через tool calls)

БД регуляций + каталог инструментов НИКОГДА не загружаются в контекстное окно. Доступ исключительно через LLM tool calls:
- `explainRegulation('art-50-1')` → текст статьи + пояснение + последствия для кода
- `searchToolDirectory('openai gpt-4o')` → профиль инструмента + статус комплаенса
- `compareJurisdictions(['eu-ai-act', 'colorado-sb205'])` → матрица сравнения
- `getProjectMemory()` → данные Уровня 1 для анализа LLM

## Обоснование

- **Системный промпт остается маленьким** (~500 токенов) вне зависимости от размера базы данных.
- **Все знания доступны контекстуально** через tool calls — LLM запрашивает то, что нужно.
- **Персистентность между сессиями**: память проекта переживает перезапуски TUI.
- **Умная суммаризация**: сессия не теряется при приближении к лимиту контекста.
- **SQLite**: полная история для аналитики и отладки.

## Последствия

- **Позитивные**: эффективное использование контекста, персистентность, масштабируемость знаний.
- **Негативные**: дополнительные tool calls = задержка для LLM-ответов, сложность суммаризации.
- **Митигация**: tool calls быстрые (чтение JSON из файла, <5мс), суммаризация только при приближении к лимиту.
