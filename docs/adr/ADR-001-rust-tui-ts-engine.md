# ADR-001: Rust TUI + TypeScript Engine

**Статус**: Принято
**Дата**: 2026-02-17
**Автор**: Marcus (CTO)

## Контекст

Complior нуждается в терминальном AI coding assistant с реалтайм-сканированием комплаенса. Рассмотрены три архитектурных варианта:

1. **Форк OpenCode** — Go TUI (BubbleTea) + TypeScript бэкенд. 90K+ звезд, проверен на масштабе.
2. **All-TypeScript (Ink)** — React-based рендеринг TUI + движок в том же процессе.
3. **Rust TUI (Ratatui) + TypeScript Engine** — клиент-сервер через HTTP/SSE.

## Решение

**Вариант 3: Rust TUI + TypeScript Engine.**

Rust-бинарник (Ratatui) рендерит TUI и обрабатывает ввод. TypeScript-процесс выполняет движок комплаенса, LLM-интеграцию и всю бизнес-логику. Коммуникация через HTTP (JSON req/resp) и SSE (стриминг LLM-токенов) на localhost.

## Обоснование

| Фактор | Форк OpenCode | TS + Ink | Rust + TS (выбрано) |
|--------|--------------|---------|---------------------|
| Запуск | ~300мс (Go) | ~500-800мс (Node) | ~10мс TUI, движок в фоне |
| Бинарник | ~25МБ (Go) | 50-100МБ (node_modules) | ~5МБ Rust |
| Качество TUI | BubbleTea (хорошо) | Ink (ограниченные виджеты) | Ratatui (лучшее: gauges, графики, canvas) |
| Рендеринг | Elm-арх., нормально | Retained mode, ре-рендер на токен | Immediate mode, суб-мс |
| Layout | Одна панель — чат | Одна панель | Split-pane: чат + сайдбар + diff |
| LLM | Их провайдер-слой | Vercel AI SDK (лучший) | Vercel AI SDK в TS engine |
| Сканер/фиксер | Нужен мост Go↔TS (IPC) | Прямой импорт (один процесс) | HTTP-вызов к TS engine (~5мс) |
| MCP Server | Их реализация | ~200 строк нового кода | TS engine stdio, ноль доп. кода |
| Контрибьюторы | Go + TS (редкая комбинация) | Только TS | Rust (TUI) + TS (engine), четкая граница |
| Upstream-риск | Каждый релиз может сломать форк | Нет | Нет |
| Потолок UX | Хороший чат | Базовый чат | Дашборд + чат + визуализация (максимум) |

### Почему не форк OpenCode?

OpenCode эволюционирует быстро (90K+ звезд). Форк означает: постоянные merge-конфликты с upstream, переписывание их инструментов, наследование их архитектурных решений. Мы берем их **паттерн** (клиент/сервер), не их код.

### Почему не all-TypeScript?

Ink (React для терминала) имеет ограниченный набор виджетов — нет графиков, sparklines, gauges. Retained mode рендеринг перерисовывает все дерево на каждый LLM-токен. Запуск 500-800мс vs 10мс для Rust.

### Почему Rust + TS — лучший вариант

- Один движок обслуживает: TUI (HTTP), MCP (stdio), VS Code расширение (HTTP), CI/CD (прямой вызов), веб-дашборд (HTTP). Пишем движок один раз — подключаем любой фронтенд.
- Rust TUI — тонкий клиент, только рендерит и принимает ввод. Если TS engine падает — TUI показывает ошибку корректно.
- Оверхед IPC минимален: скан занимает 1-3с, HTTP добавляет 5мс.
- Ratatui — самый богатый набор виджетов: gauges, sparklines, canvas, таблицы, графики.

## Последствия

- **Позитивные**: лучшее качество TUI, мгновенный запуск, малый бинарник, переиспользование движка для разных фронтендов.
- **Негативные**: два языка (Rust + TS), сложность IPC vs единый процесс, кросс-компиляция для релизов.
- **Митигация**: четкая граница — TUI рендерит, engine думает. Нет общего состояния.

## Оценки кода

- Rust TUI: ~4,000-5,000 строк
- TypeScript Engine: ~5,500-7,000 строк
- Headless CLI: ~200 строк
- Разработка: 6-8 недель для одного разработчика с AI-инструментами
